# PhotoPark - contexte agent

## Current State
Application Next.js 15 / TypeScript (`photopark`) pour inventorier, visualiser et comparer du matériel photo, avec inventaires objectifs, boîtiers et accessoires.

- **Architecture multi-pages** avec navigation. Toutes les routes authentifiées sont sous le groupe `(authenticated)/`. La racine `/` redirige vers `/lenses`.
- **Routes actuelles :**
  - `/lenses` — Liste des objectifs (tableau desktop, cartes mobile, filtres, sélection/comparaison)
  - `/lenses/chart` — Graphique focale/ouverture interactif avec zoom D3.js (molette, pan tactile, bouton réinitialiser). Toolbar (titre + "Ajouter un objectif") en pleine largeur en haut, puis `LensFiltersBar` en pleine largeur, puis zone principale : chart (4/5) à gauche, sidebar (1/5) avec liste d'objectifs filtrable uniquement (sans filtres). Deux contrôles par objectif dans la sidebar : **checkbox** (cocher/décocher, visibilité sur le graphique, sans limite) et **bouton étoile** (ajouter/retirer de la comparaison, max 5). Le graphique reçoit `checkedLenses` (filtrés des `initialLenses` par `checkedIds`, pas de limite) et `selectedIds` pour le surlignage orange des objectifs en comparaison. Clic sur une ligne/cercle du graphique → `onToggleSelected` (ajoute ou retire de la comparaison). Comparaison via popup flottant + modal.
  - `/bodies` — Inventaire boîtiers (tableau desktop, cartes mobile, filtres, détail modal, création/édition modales, sélection/comparaison)
  - `/accessories` — Inventaire **Sacs & poches** (tableau desktop, cartes mobile, filtres, détail modal, création/édition modales)
  - `/accessories/filters` — Inventaire **Filtres & bagues** dans le même domaine accessoires (tableau desktop, cartes mobile, filtres dédiés, détail modal, création/édition modales, assistant de montage)
  - `/settings/brands` — CRUD marques avec assignation de domaines `lenses` / `accessories`
   - `/settings/mounts` — CRUD montures (avec `sensorType`)
   - `/settings/options` — CRUD options objectifs (code + description + brandId), filtrées par marque sélectionnée
   - `/settings/options/groups` — CRUD groupes d'options (slug, name, type: "flag"/"value") + assignation d'options par marque
   - `/settings/accessory-types` — CRUD types d'accessoires
   - `/login` — Écran de connexion
- **Navigation :** `AppNav` (barre supérieure avec liens + bouton Déconnexion) inclut `/lenses`, `/bodies` et `/accessories`. `SubNav` est réutilisée pour `/lenses`, `/accessories` et `/settings`. Le layout `/accessories` expose deux entrées de sous-catégorie : `Sacs & poches` (`/accessories`) et `Filtres & bagues` (`/accessories/filters`).
- **État partagé via LensProvider :** Contexte React (`useLensContext`) créé dans le layout `/lenses`, qui charge les données côté serveur (`listLenses()` + `listReferenceData()`). Le contexte expose : `filters`, `setFilters`, `resetFilters`, `selectedIds`, `toggleSelected`, `clearSelection`, `hiddenIds`, `hideLens`, `editingLens`, `setEditingLens`, `showCreate`, `setShowCreate`, `initialLenses`, `filteredLenses`, `referenceData`. `filterLenses()` et `defaultFilters` sont exportés depuis `LensProvider.tsx` pour tests et usage direct.
- **État partagé via BodyProvider :** Contexte React (`useBodyContext`) créé dans le layout `/bodies`, qui charge les données côté serveur (`listBodies()` + `listBodyReferenceData()`). Le contexte expose : `filters`, `setFilters`, `resetFilters`, `selectedIds`, `toggleSelected`, `clearSelection`, `detailBody`, `setDetailBody`, `editingBody`, `setEditingBody`, `showCreate`, `setShowCreate`, `initialBodies`, `filteredBodies`, `referenceData`. `filterBodies()` et `defaultBodyFilters` sont exportés pour tests et usage direct.
- **État partagé via AccessoryProvider :** Contexte React (`useAccessoryContext`) créé dans le layout `/accessories`, qui charge les données côté serveur (`listAccessories()` + `listAccessoryReferenceData()`). Le contexte expose : `filters`, `setFilters`, `resetFilters`, `detailAccessory`, `setDetailAccessory`, `editingAccessory`, `setEditingAccessory`, `showCreate`, `setShowCreate`, `initialAccessories`, `filteredAccessories`, `referenceData`. `filterAccessories()` et `defaultAccessoryFilters` sont exportés pour tests et usage direct.
- **Persistance SQLite** via `better-sqlite3`, fichier par défaut `./data/photos.sqlite` ou `DATABASE_PATH`.
- **CRUD objectifs** avec référentiels séparés pour marques, montures et options.
- **CRUD boîtiers** avec table SQLite dédiée `bodies`, montures partagées avec les objectifs et marques partagées filtrées par domaine `bodies`.
- **CRUD accessoires** avec tables SQLite dédiées `accessory_types` et `accessories`, et référentiels partagés de marques filtrés par domaine `accessories`. Le domaine reste unique ; les sous-catégories `bag` et `filter` sont portées par `accessory_types.category`.
- Les **marques** sont partagées entre objectifs, boîtiers et accessoires via la table `brands` et une table de jointure `brand_domains`. Les sélecteurs de marques et les référentiels côté dépôt filtrent désormais par domaine (`lenses`, `bodies` ou `accessories`).
- Les options sont une relation N-N (`lens_option_links`) et portent `code` + `description` + `brandId`. Elles sont filtrées par marque dans le formulaire objectif et dans le gestionnaire de paramètres. Sur changement de marque dans le formulaire, les options ne correspondant pas à la nouvelle marque sont retirées de la sélection.
- Les **groupes d'options** (`option_groups`) permettent de classer les options par catégorie : slug, name, type (`"flag"` = Oui/Non, `"value"` = affiche le code). L'appartenance d'une option à un groupe est définie dans `option_group_members` (relation N-N). Les groupes sont optionnellement affichés dans le tableau de comparaison.
- Les montures portent le `sensorType` (`FULL_FRAME` ou `APS_C`), utilisé pour calculer l'équivalent APS-C des objectifs et partagé avec les boîtiers interchangeables.
- **Filtres :** Filtres textuel (recherche dans label, marque, monture, options), par marque, monture, option, **type d'objectif** (`kind`: `"" | "prime" | "zoom"` via un select `Tous` / `Fixe` / `Zoom`), statut (favoris/à acheter/possédés), et quatre plages numériques via `DualRangeSlider` : focale min, focale max, ouverture à focale min, ouverture à focale max. Les plages utilisent le type `LensFilters` avec des champs `*Low`/`*High` (nombres), vérifiés de manière inclusive par `filterLenses()`. `filterLenses()` applique le filtre `Fixe` avec `isPrimeLens()` et `Zoom` comme inverse. Les sliders de focale utilisent désormais une plage UI **0–300** (plus 0–1000) ; la borne haute `300` est **ouverte** et signifie `300 mm et plus`, affichée `300+ mm` uniquement pour la valeur haute. La borne basse reste affichée comme une valeur simple (`300 mm`). Valeurs par défaut : focales [0, 300], ouvertures [1, 30]. Plus de filtre `sensorType`.
- **UI inventaire :** Tableau desktop, cartes mobile, graphique focale/ouverture, sélection et comparaison de 2 à 5 objectifs. Le tableau et les cartes affichent le type d'objectif dérivé (`Fixe` si `focalMinMm === focalMaxMm`, sinon `Zoom`), les badges de statut (`Favori`, `À acheter`, `Possédé`, `Retiré`) et évitent de répéter les plages identiques (ex. `7.8 mm`, `f/4`). Les prix et poids sont formatés de façon cohérente dans la liste, les cartes et la comparaison : `formatPrice()` groupe les milliers avec des espaces et affiche toujours 2 décimales ; `formatWeight()` groupe les milliers avec des espaces sans décimales. La valeur `0` est considérée comme valide et doit être affichée, pas traitée comme vide.
- **UI boîtiers :** inventaire boîtiers en tableau desktop et cartes mobile, avec filtres par texte, marque, monture, format capteur, type (`mirrorless`/`dslr`) et statut. Le détail s'affiche dans `BodyDetailModal`; la création et l'édition utilisent `BodyForm` en modal. Les boîtiers portent : marque, nom/libellé, type, flag objectifs interchangeables, format capteur, monture partagée (nullable si boîtier à objectif fixe), mégapixels, ISO min/max, prix, poids, rafale, vidéo, autonomie, flags IBIS/double slot/tropicalisation/écran orientable, notes, statuts (`Favori`, `À acheter`, `Possédé`, `Retiré`).
- **Comparaison boîtiers :** sélection de 2 à 5 boîtiers via `toggleSelected` du contexte ; `BodyComparePopup` affiche la sélection en bas et ouvre une modale contenant `BodyCompareTable`.
- **UI accessoires :** le domaine `/accessories` est scindé en deux sous-catégories. `Sacs & poches` garde le tableau desktop/cartes mobile, les filtres texte/marque/type/statut/laptop/trépied, le détail via `AccessoryDetailModal` et la création/édition via `AccessoryForm`. `Filtres & bagues` réutilise les mêmes actions/provider/référentiels, mais avec des filtres dédiés (texte, marque, type, statut, localisation, liaison), un formulaire spécialisé et un `FilterAssemblyAssistant` interactif en pile visuelle (Pillar Stack) : sélecteur d'objectif filtrant pour `(isOwned || isFavorite || isNextPurchase) && !retired` avec `useEffect` synchronisant `lensId`, puis une pile verticale de blocs (`LensBlock` en bas, `ConnectorLine` trait plein/pontillé selon fileté/magnétique, `AccessoryBlock` coloré par rôle avec bouton × pour démonter, `AddSlot` en haut avec bouton + ouvrant une liste de pièces compatibles et bouton "Monter") et une illustration SVG latérale `FilterStackIllustration` (style dessin d'enfant, rendu par rôle : bagues filetée/magnétique, disque filtre, pare-soleil, bloc générique), le tout dans un layout flex horizontal `.filter-stack-layout`. Calcule tout en interne via `getActiveMountInterface`, `getMountedAssembly`, `getMountableAccessories`. `AccessoryLensReference` porte désormais les flags `isOwned`, `isFavorite`, `isNextPurchase`, `retired` peuplés depuis `listLenses()` dans `listAccessoryReferenceData()`. Pour les rôles non `general`, le formulaire ne doit pas persister un nom/type libre : il dérive l'intitulé affiché depuis le rôle et les interfaces. Le formulaire accessoire soumet désormais explicitement le `<form>` via `onSubmit` + `new FormData(event.currentTarget)` ; en cas d'échec serveur/validation, les valeurs saisies restent en place et les erreurs sérialisées sont reformattées en texte lisible. Le composant interne `Field` alterne entre mode contrôlé (`value` + `onChange` si `value` fournie) et non contrôlé (`defaultValue` sinon) pour éviter l'avertissement React `defaultValue`+`value` simultanés.
- **Accessoires filtres/bagues :** les pièces de sous-catégorie `filter` portent `storageLocation`, montage direct sur objectif ou sur accessoire parent, interfaces arrière/avant (`none` / `threaded` / `magnetic`), rôle (`general` / `filter` / `adapter` / `hood`), `filterStrength` et `supportsMagneticHood`. Pour le montage magnétique, le diamètre est utilisé directement (plus de correspondance via nom de système). Pour les adaptateurs, le couple rôle/interfaces pilote désormais aussi le type et le nom dérivés : `threaded→threaded` avec diamètres différents = `Bague de conversion`, `threaded→magnetic` avec diamètres différents = `Bague de réduction magnétique`, `threaded→magnetic` avec même diamètre = `Bague magnétique`, avec suffixe `avec pare-soleil` si le support du pare-soleil magnétique est coché. Les combinaisons `threaded→threaded` à diamètre identique, `magnetic→magnetic` et `magnetic→threaded` sont invalides. Les montages sont validés côté écriture pour empêcher les combinaisons incohérentes.
- Les objectifs ont un booléen `retired` (badge/libellé UI `Retiré`). Le formulaire objectif propose un import/aperçu client depuis un libellé puis expose aussi une case à cocher `Retiré` ; les marques, montures, focales, ouvertures max et options reconnues sont préremplies depuis les référentiels existants, puis l'aperçu est régénéré depuis les champs structurés. Après création ou modification réussie, le formulaire se ferme automatiquement ; il reste ouvert si l'action serveur échoue (ex. doublon détecté), et l'erreur est affichée dans un `<div className="form-error" role="alert">`.
- Les pages de paramètres (marques, montures, options, types d'accessoires) chargent leurs propres données référentielles côté serveur — elles n'utilisent ni `LensProvider`, ni `BodyProvider`, ni `AccessoryProvider`.
- Tests Vitest sur validation, utilitaires métier, parsing de libellés, dépôts SQLite, formulaires, page chart et popups/tableaux de comparaison. Côté accessoires : `AccessoryForm.test.tsx` couvre le rendu des modes contrôlé/non contrôlé du composant interne `Field` ; `FilterAssemblyAssistant.test.ts` couvre le filtrage du sélecteur d'objectif (retirés exclus, non-flag exclus, next-purchase inclus). Côté boîtiers : validation, utilitaires `body-utils`, dépôt SQLite, `filterBodies()` (`BodyProvider.test.ts`), `BodyCompareTable`, ainsi que validation des domaines partagés et assertions `BrandManager` pour le domaine `bodies`.

## Usage
- **Pour tout affichage lié aux objectifs :** utiliser `useLensContext()` depuis `@/components/lens/LensProvider` pour accéder aux filtres, sélection, masquage et données. Ne pas dupliquer l'état.
- **Pour tout affichage lié aux boîtiers :** utiliser `useBodyContext()` depuis `@/components/body/BodyProvider` pour accéder aux filtres, sélection, modales et données. Ne pas dupliquer l'état.
- **Pour tout affichage lié aux accessoires :** utiliser `useAccessoryContext()` depuis `@/components/accessory/AccessoryProvider` pour accéder aux filtres, modales détail/édition/création et données. Ne pas dupliquer l'état.
- **Pour créer/modifier/supprimer un objectif :** utiliser les actions serveur dans `src/app/actions/lens-actions.ts` (`createLensAction`, `updateLensAction`, `deleteLensAction`). Elles appliquent auth, CSRF et `revalidatePath("/lenses", "layout")`. Une vérification de doublon est effectuée dans la transaction côté dépôt : si un objectif avec les mêmes `brandId`, `focalMinMm`, `focalMaxMm`, `maxApertureAtMinFocal`, `maxApertureAtMaxFocal`, `mountId` existe déjà, un `DuplicateLensError` est levé (message contenant le libellé existant). Exception actuelle : `updateLens()` autorise les modifications de champs non identitaires sur une ligne legacy déjà dupliquée, tant que cette mise à jour conserve exactement les mêmes champs d'identité ; en revanche, toute modification des champs d'identité qui entrerait en collision avec une autre ligne reste rejetée. L'erreur est attrapée par le formulaire client (`submitAndClose` try/catch) et affichée dans le `<div className="form-error">` — le formulaire reste ouvert pour permettre à l'utilisateur de corriger ou annuler.
- **Pour créer/modifier/supprimer un boîtier :** utiliser `src/app/actions/body-actions.ts` (`createBodyAction`, `updateBodyAction`, `deleteBodyAction`). Elles appliquent auth, CSRF et revalident `/bodies`.
- **Pour créer/modifier/supprimer un accessoire ou un type d'accessoire :** utiliser `src/app/actions/accessory-actions.ts` (`createAccessoryAction`, `updateAccessoryAction`, `deleteAccessoryAction`, `createAccessoryTypeAction`, `updateAccessoryTypeAction`, `deleteAccessoryTypeAction`). Elles appliquent auth, CSRF, et revalident le layout `/accessories` partagé par `/accessories` et `/accessories/filters`, ainsi que la page de paramètres concernée pour les types.
- **Pour monter/démonter un accessoire filtre/bague :** utiliser `mountAccessoryOnLensAction(accessoryId, lensId)`, `mountAccessoryOnAccessoryAction(accessoryId, parentAccessoryId)` ou `unmountAccessoryAction(accessoryId)` depuis `src/app/actions/accessory-actions.ts`. Elles retournent `{ success, error }`, appliquent `assertAuthenticated()`, et revalident `/accessories`.
- **Pour créer/modifier/supprimer une marque, monture, option ou groupe d'options :** utiliser les actions correspondantes (`createBrandAction`, `updateMountAction`, `deleteOptionAction`, `createOptionGroupAction`, `setOptionGroupMembersAction`, etc.). Les actions de marques revalident `/settings/brands`, `/lenses`, `/bodies` **et** `/accessories` car les marques sont partagées entre domaines. Les montures sont partagées entre objectifs et boîtiers.
- **Les pages de paramètres** (`/settings/brands`, `/settings/mounts`, `/settings/options`, `/settings/options/groups`, `/settings/accessory-types`) chargent leurs propres données référentielles côté serveur — elles n'utilisent ni `LensProvider`, ni `BodyProvider`, ni `AccessoryProvider`.
- **Le zoom D3.js dans LensChart :** navigation zoom (pas loupe). Les traits utilisent `vectorEffect="non-scaling-stroke"` pour une épaisseur constante. Les rayons des cercles sont contre-échelonnés via la fonction `counterScaleCircles(root, k)`. Le zoom/pan est initialisé une seule fois dans un `useEffect(() => {}, [])` ; la transformation D3 est appliquée directement au DOM — aucune re-render React pendant les gestes. L'instance D3 zoom est stockée dans `zoomRef`. `counterScaleCircles` est appelée depuis le handler D3 (tous les événements de zoom) et depuis un `useLayoutEffect` qui re-synchronise en amont du rendu quand `lenses` ou `selectedIds` changent (pour éviter un flash de rayons non contre-échelonnés après un re-render React en zoom). Ne pas remplacer ce pattern par un état React.
- **Contre-échelle des cercles :** une seule constante contrôle le comportement : `MIN_CIRCLE_R` (2). La formule appliquée est `Math.max(MIN_CIRCLE_R, base / k)` avec un plancher. Les cercles restent à une taille visuelle constante à tout niveau de zoom (rayon visuel = `k × (base/k)` = **base constant**). Les cercles portent un attribut `data-base-r` contenant le rayon de base. Pour modifier le rendu du zoom, éditer le `MIN_CIRCLE_R` et/ou `base / k` dans `counterScaleCircles`.
- **Le graphique montre uniquement les objectifs cochés :** `LensChart` reçoit `checkedLenses` (filtrés depuis `initialLenses` par `checkedIds`, état local `useState` dans `LensChartPage`, pas de limite). L'état initial coche uniquement les objectifs possédés et non retirés (`isOwned && !retired`) ; les autres restent disponibles dans la sidebar et peuvent être cochés manuellement. Les filtres (`LensFiltersBar`) et `hiddenIds` n'affectent que la liste de la sidebar (pour faciliter le choix), pas le jeu d'objectifs disponibles pour la sélection. Le clic sur une ligne/cercle du graphique appelle `onToggleSelected` (ajoute ou retire de la comparaison, max 5). Les objectifs en comparaison sont surlignés en orange via `selectedIds`. La sidebar a deux contrôles par ligne : **checkbox** (`toggleChecked`, visibilité, pas de limite) et **bouton étoile** (`toggleSelected`, comparaison, max 5). Le clic sur le libellé déclenche aussi `toggleChecked`.
- **Bouton Réinitialiser :** utiliser `resetZoom()` (mémorisée via `useCallback`) pour une transition fluide de 500ms vers `d3.zoomIdentity`. Le bouton est toujours rendu, même dans l'état vide.
- **Filtres plages numériques :** réutiliser `DualRangeSlider` (props : `min`, `max`, `low`, `high`, `onChange`, `step`, `label`, `formatValue`, avec `formatLowValue` / `formatHighValue` optionnels si les deux bornes doivent être affichées différemment). Les contraintes `low ≤ high` sont assurées par les handlers `handleLowChange`/`handleHighChange` qui clampent au moment du changement. Pour la focale, réutiliser les constantes partagées de `src/lib/lens/filter-constants.ts` et conserver la sémantique actuelle : borne haute `300` affichée `300+ mm`, borne basse inchangée (`300 mm`).
- **`filterLenses()` (pure function) :** importable depuis `LensProvider.tsx` pour tests et usage direct. Vérifie chaque filtre de manière inclusive : la valeur de l'objectif doit être entre `*Low` et `*High` inclus. Le filtre `kind` accepte `prime` / `zoom` et s'appuie sur `isPrimeLens()` (zoom = inverse). Les champs `sensorType`, `focalMin`, `focalMax`, `maxAperture` ont été supprimés de `LensFilters` ; utiliser `kind` et les paires `*Low`/`*High` à la place.
- Réutiliser `lens-repository.ts` pour toute lecture/écriture SQLite au lieu d'accéder directement à la base depuis les composants.
- Réutiliser `body-repository.ts` pour toute lecture/écriture SQLite liée aux boîtiers. Les labels sont dérivés côté dépôt via `normalizeBodyInput()` / `generateBodyLabel()` ; ne pas persister de libellé libre.
- Réutiliser `accessory-repository.ts` pour toute lecture/écriture SQLite liée aux accessoires. Les labels sont dérivés côté dépôt via `normalizeAccessoryInput()` / `generateAccessoryLabel()` ; ne pas persister de libellé libre.
- Réutiliser `parseLensFormData` / `lensSchema` pour le formulaire objectif, y compris le booléen `retired` ; les nombres peuvent utiliser un point ou une virgule.
- Réutiliser `parseBodyFormData` / `bodySchema` pour le formulaire boîtier ; les nombres peuvent utiliser un point ou une virgule, les champs vides deviennent `null`, `mountId` est requis seulement si `isInterchangeableLens === true`.
- Réutiliser `parseAccessoryFormData` / `accessorySchema` pour le formulaire accessoire ; les nombres peuvent utiliser un point ou une virgule, et les champs vides deviennent `null`. Pour les accessoires `filter` dont le rôle n'est pas `general`, laisser le serveur dériver `typeId` et le nom persisté depuis le rôle, les interfaces et les diamètres au lieu de forcer une saisie libre. Conserver le pattern actuel de soumission explicite `onSubmit` → `new FormData(event.currentTarget)` dans `AccessoryForm` pour que les échecs de soumission n'effacent pas l'état local et que les erreurs sérialisées soient affichées dans un format lisible. Le composant interne `Field` alterne entre mode contrôlé (`value` + `onChange` si `value` fournie) et non contrôlé (`defaultValue` sinon) : réutiliser ce pattern pour tout champ `<input>` d'AccessoryForm afin d'éviter l'avertissement React lorsque les deux props sont présentes simultanément.
- Pour créer/modifier des objectifs, fournir des IDs de référentiels (`brandId`, `mountId`, `optionIds`) ; marque, monture, capteur, options, label et équivalents APS-C sont normalisés côté dépôt.
- Pour importer un objectif depuis un libellé collé, réutiliser `parseLensLabel` avec les référentiels chargés ; pour les libellés générés/aperçus, réutiliser `generateLensLabel`. Ne pas persister de libellé libre : le serveur dérive toujours `label` depuis les champs structurés. Le parser filtre les options par marque identifiée (`parsed.brandId`) pour éviter les collisions entre marques.
- **Options par marque :** Les options sont liées à une marque spécifique via `brandId`. Dans le formulaire objectif (`LensForm.tsx`), `brandOptions` est un `useMemo` filtrant `referenceData.options` par `brandId` sélectionné. Sur changement de marque, les options ne correspondant pas à la nouvelle marque sont retirées de la sélection (`updateFields`). Dans le gestionnaire de paramètres (`OptionManager.tsx`), un sélecteur de marque filtre les options affichées.
- **Marques par domaine :** réutiliser `listBrandsByDomain()` / `listBrandsWithDomains()` et la validation `parseBrandFormData()` plutôt que de filtrer à la main. Les formulaires objectifs doivent proposer uniquement les marques `lenses`, les formulaires boîtiers uniquement les marques `bodies`, les formulaires accessoires uniquement les marques `accessories`. `updateBrand()` refuse de retirer un domaine encore utilisé par des lignes existantes (`lenses`, `bodies` ou `accessories`).
- **Filtres boîtiers :** réutiliser `filterBodies()` (pure function) exportée par `BodyProvider.tsx`. Conserver l'ensemble actuel de filtres UI : `query`, `brand`, `mount`, `sensorFormat`, `bodyType`, `status`.
- **UI boîtiers :** utiliser `BodyDetailModal` pour l'affichage détaillé et `BodyForm` pour création/édition. Le flux existant ferme le détail avant d'ouvrir l'édition ; conserver ce pattern pour éviter des modales concurrentes.
- **Filtres accessoires :** réutiliser `filterAccessories()` (pure function) exportée par `AccessoryProvider.tsx`. Les filtres disponibles dépendent de `typeCategory` : `bag` utilise `query`, `brand`, `type`, `status`, `laptop`, `tripod` ; `filter` utilise `query`, `brand`, `type`, `status`, `location`, `mountType`. Réutiliser `sanitizeAccessoryFilters()` lors d'un changement de sous-catégorie pour éviter que des champs masqués fuient d'une page à l'autre.
- **UI accessoires :** traiter `Filtres & bagues` comme une sous-catégorie du domaine accessoires, pas comme un nouveau domaine autonome. Réutiliser `AccessoryDetailModal`, `AccessoryForm`, `AccessoryProvider`, les actions serveur et `accessory-repository.ts`, puis appliquer la sémantique spécifique via `typeCategory`.
- **Le `FilterAssemblyAssistant`** est automatiquement inclus par `AccessoryListPage` pour la sous-catégorie `filter`. Il ne reçoit plus de prop `compatibility` : il calcule la pile via `getActiveMountInterface`, `getMountedAssembly` et `getMountableAccessories` depuis `filter-compatibility.ts`, et utilise `mountAccessoryOnLensAction`, `mountAccessoryOnAccessoryAction`, `unmountAccessoryAction` pour les interactions. Affiche la pile interactive (Pillar Stack) côte à côte avec l'illustration SVG `FilterStackIllustration` dans un layout flex horizontal `.filter-stack-layout`. Si un composant tiers a besoin de données de compatibilité, réutiliser `getLensAccessoryCompatibility()` depuis `filter-compatibility.ts` (retourne `mounted`, `available`, `moveRequired`).
- **Montages filtres/bagues :** réutiliser `getLensAccessoryCompatibility()` pour les suggestions de l'`AccessoryDetailModal`, `getActiveMountInterface()` pour l'interface active en bout de chaîne, `getMountedAssembly()` pour la chaîne montée sur un objectif, `getMountableAccessories()` pour les pièces compatibles avec une interface donnée, et `matchesMount()` pour la vérification unitaire. Côté écriture, utiliser les actions serveur `mountAccessoryOnLensAction`, `mountAccessoryOnAccessoryAction`, `unmountAccessoryAction` (ou les méthodes du dépôt directement) pour les opérations de montage/démontage, et `assertAccessoryMountConsistency()` pour les validations d'écriture. Pour les adaptateurs, conserver les règles actuelles de dérivation/validation : seuls `threaded→threaded` (diamètres différents) et `threaded→magnetic` (même ou différents diamètres) sont acceptés ; les autres combinaisons restent rejetées. La validation applicative bloque déjà les montages incohérents ; des suivis d'intégrité non bloquants restent connus autour de la revalidation des descendants et d'une unicité renforcée au niveau base de données.
- **Groupes d'options :** CRUD via `/settings/options/groups` (composant `OptionGroupManager.tsx`). Chaque groupe a un `slug`, `name`, et `type` (`"flag"` ou `"value"`). Les options sont assignées aux groupes par marque via `setOptionGroupMembersAction` (appelle `replaceGroupMembers`). L'affichage dans la comparaison (`LensCompareTable`) utilise les props optionnelles `optionGroups` et `optionGroupMembers` pour ajouter des lignes par groupe : `flag` → "Oui"/"—", `value` → codes séparés par virgule.
- Pour la comparaison : sélection de 2 à 5 objectifs via `toggleSelected` du contexte ; utiliser `LensComparePopup` (qui contient `LensCompareTable` dans sa modale). La popup et la table acceptent optionnellement les props `optionGroups` et `optionGroupMembers` pour afficher les lignes de groupes d'options dans la comparaison.
- Pour la comparaison boîtiers : sélection de 2 à 5 boîtiers via `toggleSelected` du contexte ; utiliser `BodyComparePopup` (qui contient `BodyCompareTable` dans sa modale).
- Pour les calculs optiques, libellés, plages focale/ouverture affichées, type `Fixe`/`Zoom` et formatage prix/poids, réutiliser `src/lib/lens/lens-utils.ts` ; le filtre `kind` doit continuer à réutiliser `isPrimeLens()` plutôt que re-déduire la logique dans les composants. Ne dupliquer ni le facteur APS-C, ni la génération de label, ni les règles d'affichage des plages identiques, ni le formatage numérique. Pour prix/poids, ne pas tester la présence avec un simple check truthy : `0` doit rester affiché.
- Pour les labels, libellés dérivés (`Hybride`/`Reflex`, formats capteur), booléens d'équipement et formatage prix/poids/ISO/rafale/autonomie des boîtiers, réutiliser `src/lib/body/body-utils.ts`. `formatBodyPrice()` et `formatBodyWeight()` doivent continuer à afficher `0` comme valeur valide.
- Pour étendre l'app à d'autres familles que sacs/poches : suivre le pattern accessoires existant (layout dédié + provider + filtres + tableau/cartes + modal détail + modal formulaire + dépôt SQLite dédié).

## Notes
- **Layout de la page chart :** `LensChartPage` utilise `.chart-page-layout` (flex horizontal), `.chart-page-main` (flex: 4) pour la zone graphique, `.chart-page-sidebar` (flex: 1) pour la liste à cocher uniquement. La toolbar et les filtres sont en pleine largeur au-dessus (class `.chart-page-full`). La comparaison s'affiche via `LensComparePopup` (flottant fixe en bas).
- **CSS sidebar chart :** `.chart-lens-item.checked` (fond bleu pour objectifs visibles), `.chart-lens-item.checked.selected` (fond ambre quand coché ET en comparaison), `.chart-lens-compare-btn` / `.chart-lens-compare-btn.active` (bouton étoile).
- **Alignement numérique inventaire :** `LensTable` utilise `.numeric-cell` pour aligner à droite les cellules prix/poids ; `LensCard` utilise `.numeric-value` pour aligner la valeur du prix. Conserver ces classes pour toute nouvelle valeur monétaire/pondérale affichée dans l'inventaire.
- **Accessoires et formatage :** `AccessoryTable` réutilise `.numeric-cell` pour poids/prix et `AccessoryCard` réutilise `.numeric-value` pour le prix. Les helpers `formatAccessoryPrice()` / `formatAccessoryWeight()` délèguent à `formatPrice()` / `formatWeight()` et doivent continuer à afficher `0` comme valeur valide. Pour les filtres-bagues, réutiliser aussi `formatAccessoryLocation()`, `formatAccessoryInterface()` et `formatMountEndpoint()` (sans paramètre `system` ; pour le magnétique, le diamètre est directement utilisé).
- **CSS FilterStackLayout :** le layout flex horizontal de l'assistant de montage utilise `.filter-stack-layout` (conteneur parent), `.filter-stack-illustration-wrap` (wrapper de l'illustration SVG) et `.filter-stack-svg` (élément SVG). Ces classes sont définies dans `globals.css`.
- **Boîtiers et monture fixe :** `BodyForm` désactive le select de monture quand `isInterchangeableLens` est décoché, et le dépôt force alors `mountId`/`mount` à `null`. Dans la comparaison, un boîtier non interchangeable affiche `Fixe` sur la ligne `Monture`.
- **Filtres et sidebar :** les filtres (`LensFiltersBar`) et `hiddenIds` (via `hideLens`) n'affectent que la liste de la sidebar — pas le graphique. Le graphique reçoit `initialLenses` et montre uniquement les objectifs cochés (`checkedLenses` filtré par `checkedIds`). Au chargement, `checkedIds` est initialisé avec les objectifs possédés et non retirés. Le clic sur une ligne/cercle du graphique appelle `onToggleSelected` (ajoute/retire comparaison). Les checkboxes de la sidebar contrôlent la visibilité sur le graphique. Les boutons étoile contrôlent la comparaison. Contrairement aux pages liste, la sidebar de la page chart ne contient **pas** les filtres — ils sont en pleine largeur au-dessus.
- **Sémantique focale 300+ :** la borne haute des sliders de focale est volontairement ouverte. `300` dans `focalMinHigh` / `focalMaxHigh` signifie `>= 300 mm`, mais cette notation `+` ne doit être montrée que pour les labels/affichages de borne haute ; les bornes basses gardent un libellé normal.
- L'ouverture max à focale max est optionnelle côté formulaire ; la validation remplace une valeur vide par l'ouverture max à focale min.
- `minAperture` doit rester supérieure ou égale aux ouvertures maximales, avec fallback sur l'ouverture max à focale min si celle à focale max est vide.
- `initializeSchema()` migre automatiquement l'ancienne table `lens_options` sans `brandId` : renommage, recréation avec `UNIQUE(code, brandId)`, assignation des options orphelines à la marque "Autre". Répare aussi `lens_option_links` si sa FK `lensId` pointe encore vers une table legacy `lenses_legacy_*`. En complément, `ensureLensColumns(database)` ajoute au démarrage les colonnes manquantes de `lenses` (`minApertureAtMinFocal`, `minApertureAtMaxFocal`, `retired`) sur les bases SQLite existantes ; il faut redémarrer `npm run dev` pour déclencher ces réparations au démarrage du dépôt.
- Le schéma SQLite partagé inclut désormais `brand_domains` pour affecter une marque à `lenses`, `bodies`, `accessories` ou plusieurs domaines. `getDatabase()` et `initializeSchema()` injectent aussi rétroactivement le domaine `lenses` pour les marques historiques et reconstruisent la contrainte SQL de `brand_domains` pour accepter `bodies`.
- Une migration legacy renomme l'ancienne table `lenses` en `lenses_legacy_<timestamp>` puis recrée le schéma actuel et transforme les anciennes chaînes `brand`, `mount`, `options` en référentiels.
- `seedReferenceData` crée les référentiels Canon : marque, montures (RF, RF-S, EF, EF-S), options (L, IS, USM, STM) avec `brandId`, et 3 groupes d'options (Stabilisation/flag, Motorisation/value, Série/value) avec leurs membres.
- `body-repository.ts` crée au démarrage la table `bodies` dans la même base SQLite partagée, et `listBodyReferenceData()` réutilise `listBrandsByDomain("bodies")` + les montures partagées.
- `accessory-repository.ts` crée au démarrage les tables `accessory_types` et `accessories`, applique d'abord les réparations/migrations de colonnes legacy des accessoires, puis crée les index SQLite avant le seed des types par défaut `bag` (`Sac à dos`, `Bandoulière`, `Poche`, `Ceinture`) et `filter` (`Filtre`, `Bague de conversion`, `Bague de réduction magnétique`, `Bague magnétique`, `Adaptateur / pare-soleil magnétique`). Cet ordre évite les échecs de démarrage sur des bases locales mises à niveau (`no such column: mountedOnLensId`).
- `LensChart.tsx` utilise des constantes module-level (`CHART_WIDTH`, `CHART_HEIGHT`, `MARGIN`, `PALETTE`, `ZOOM_SCALE_EXTENT`, `MIN_CIRCLE_R`) pour la configuration du graphique et du zoom.
- Le `useEffect` de zoom D3 nettoie ses écouteurs au démontage : `svg.on(".zoom", null)` pour l'SVG et `d3.select(window).on(".zoom", null)` pour les écouteurs gestuels au niveau fenêtre (pinch-to-zoom).
- `import React from "react"` est requis pour le transform JSX classique utilisé par Vitest (le mock de `react` dans les tests l'exige).
- Variables importantes : `SESSION_SECRET` (32 caractères minimum), `APP_PASSWORD_HASH` de préférence, fallback `APP_PASSWORD`, `DATABASE_PATH`, `TRUST_PROXY`, `NEXT_PUBLIC_APSC_CROP_FACTOR`.
- Le cookie de session s'appelle `photos_session`, est signé HMAC SHA-256 et expire après 14 jours.
- Le dépôt Git n'a pas de remote connu dans le contexte fourni. Rappel demandé : vérifier avec `git remote -v` ; ajouter un remote avec `git remote add origin <url>` ; préparer un commit avec `git status`, `git add <fichiers>`, `git commit -m "message"`, puis pousser avec `git push -u origin <branche>`.

- `npm run dev` — développement Next.js ; à redémarrer pour déclencher init/migration/repair SQLite.
- `npm run build` — build production.
- `npm run start` — serveur production après build.
- `npm run typecheck` — TypeScript sans émission.
- `npm run test` — suite Vitest.
