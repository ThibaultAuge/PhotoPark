# PhotoPark - contexte agent

## Current State
Application Next.js 15 / TypeScript (`photopark`) pour inventorier, visualiser et comparer du matériel photo, avec inventaires objectifs, boîtiers et accessoires.

- **Architecture multi-pages** avec navigation. Toutes les routes authentifiées sont sous le groupe `(authenticated)/`. La racine `/` redirige vers `/lenses`.
- **Routes actuelles :**
  - `/lenses` — Liste des objectifs (tableau desktop, cartes mobile, filtres, sélection/comparaison)
  - `/lenses/chart` — Graphique focale/ouverture interactif avec zoom D3.js (molette, pan tactile, bouton réinitialiser). Toolbar (titre + "Ajouter un objectif") en pleine largeur en haut, puis `LensFiltersBar` en pleine largeur, puis zone principale : chart (4/5) à gauche, sidebar (1/5) avec liste d'objectifs filtrable uniquement (sans filtres). Deux contrôles par objectif dans la sidebar : **checkbox** (cocher/décocher, visibilité sur le graphique, sans limite) et **bouton étoile** (ajouter/retirer de la comparaison, max 5). Le graphique reçoit `checkedLenses` (filtrés des `initialLenses` par `checkedIds`, pas de limite) et `selectedIds` pour le surlignage orange des objectifs en comparaison. Clic sur une ligne/cercle du graphique → `onToggleSelected` (ajoute ou retire de la comparaison). Comparaison via popup flottant + modal.
  - `/bodies` — Inventaire boîtiers (tableau desktop, cartes mobile, filtres, détail modal, création/édition modales, sélection/comparaison)
  - `/accessories` — Inventaire **Sacs & poches** (tableau desktop, cartes mobile, filtres, détail modal, création/édition modales)
  - `/accessories/filters` — Inventaire **Filtres & bagues** dans le même domaine accessoires (tableau desktop, cartes mobile, filtres dédiés, détail modal, création/édition modales, assistant de montage)
  - `/accessories/others` — Inventaire **Autres accessoires** dans le même domaine accessoires (tableau desktop, cartes mobile, filtres dédiés, détail modal, création/édition modales, champs conditionnels par profil)
  - `/settings/brands` — CRUD marques avec assignation de domaines `lenses` / `accessories`
   - `/settings/mounts` — CRUD montures (avec `sensorType`)
   - `/settings/options` — CRUD options objectifs (code + description + brandId), filtrées par marque sélectionnée
   - `/settings/options/groups` — CRUD groupes d'options (slug, name, type: "flag"/"value") + assignation d'options par marque
   - `/settings/accessory-types` — CRUD types d'accessoires
   - `/login` — Page serveur de connexion ; redirige directement vers `/lenses` si `hasValidSession()` est déjà vrai, sinon rend le client `LoginForm`
- **Navigation :** `AppNav` (barre supérieure avec liens + bouton Déconnexion) inclut `/lenses`, `/bodies` et `/accessories`. `SubNav` est réutilisée pour `/lenses`, `/accessories` et `/settings`. Le layout `/accessories` expose trois entrées de sous-catégorie : `Sacs & poches` (`/accessories`), `Filtres & bagues` (`/accessories/filters`) et `Autres accessoires` (`/accessories/others`).
- **État partagé via LensProvider :** Contexte React (`useLensContext`) créé dans le layout `/lenses`, qui charge les données côté serveur (`listLenses()` + `listReferenceData()`). Le contexte expose : `filters`, `setFilters`, `resetFilters`, `selectedIds`, `toggleSelected`, `clearSelection`, `hiddenIds`, `hideLens`, `editingLens`, `setEditingLens`, `showCreate`, `setShowCreate`, `initialLenses`, `filteredLenses`, `referenceData`. `filterLenses()` et `defaultFilters` sont exportés depuis `LensProvider.tsx` pour tests et usage direct.
- **État partagé via BodyProvider :** Contexte React (`useBodyContext`) créé dans le layout `/bodies`, qui charge les données côté serveur (`listBodies()` + `listBodyReferenceData()`). Le contexte expose : `filters`, `setFilters`, `resetFilters`, `selectedIds`, `toggleSelected`, `clearSelection`, `detailBody`, `setDetailBody`, `editingBody`, `setEditingBody`, `showCreate`, `setShowCreate`, `initialBodies`, `filteredBodies`, `referenceData`. `filterBodies()` et `defaultBodyFilters` sont exportés pour tests et usage direct.
- **État partagé via AccessoryProvider :** Contexte React (`useAccessoryContext`) créé dans le layout `/accessories`, qui charge les données côté serveur (`listAccessories()` + `listAccessoryReferenceData()`). Le contexte expose : `filters`, `setFilters`, `resetFilters`, `detailAccessory`, `setDetailAccessory`, `editingAccessory`, `setEditingAccessory`, `showCreate`, `setShowCreate`, `initialAccessories`, `filteredAccessories`, `referenceData`. Le provider garde désormais l'état modal/création/édition cohérent lors d'un changement de sous-catégorie (`bag` / `filter` / `other`) en réinitialisant les états incompatibles. Les filtres partagés accessoires n'incluent plus `compatibleLensId` ni `onlyCompatible`. `filterAccessories()` et `defaultAccessoryFilters` sont exportés pour tests et usage direct, mais `AccessoryListPage` re-sanitize maintenant les filtres selon `typeCategory` et recalcule localement le rendu filtré depuis `initialAccessories` au lieu de dépendre directement de `filteredAccessories` pour l'affichage.
- **Persistance SQLite** via `better-sqlite3`, fichier par défaut `./data/photos.sqlite` ou `DATABASE_PATH`.
- **Déploiement conteneur** : le flux recommandé est **GitHub Actions -> GHCR -> Portainer via pull d'image**. `publish-image.yml` publie toujours un tag `sha-<commit>` et ne publie `latest` que pour les pushes sur `main`; une `concurrency` par ref empêche une ancienne exécution d'écraser `latest`. `docker-compose.portainer.yml` est le compose de production dédié pour Portainer et consomme `PHOTOPARK_IMAGE` (préférer un tag SHA ; un digest est encore plus fort qu'un tag). `docker-compose.yml` reste limité aux usages locaux/manuels en mode `build: .`. Il n'existe plus de workflow `deploy.yml` ni de wrapper `scripts/deploy.sh` : les déploiements manuels passent par des commandes `docker compose` standard.
- **CRUD objectifs** avec référentiels séparés pour marques, montures et options.
- **CRUD boîtiers** avec table SQLite dédiée `bodies`, montures partagées avec les objectifs et marques partagées filtrées par domaine `bodies`.
- **CRUD accessoires** avec tables SQLite dédiées `accessory_types` et `accessories`, et référentiels partagés de marques filtrés par domaine `accessories`. Le domaine reste unique ; les sous-catégories `bag`, `filter` et `other` sont portées par `accessory_types.category`. Les types `other` peuvent porter un `profile` stable optionnel pour piloter les champs UI et la persistance des specs dédiées.
- Les **marques** sont partagées entre objectifs, boîtiers et accessoires via la table `brands` et une table de jointure `brand_domains`. Les sélecteurs de marques et les référentiels côté dépôt filtrent désormais par domaine (`lenses`, `bodies` ou `accessories`).
- Les options sont une relation N-N (`lens_option_links`) et portent `code` + `description` + `brandId`. Elles sont filtrées par marque dans le formulaire objectif et dans le gestionnaire de paramètres. Sur changement de marque dans le formulaire, les options ne correspondant pas à la nouvelle marque sont retirées de la sélection.
- Les **groupes d'options** (`option_groups`) permettent de classer les options par catégorie : slug, name, type (`"flag"` = Oui/Non, `"value"` = affiche le code). L'appartenance d'une option à un groupe est définie dans `option_group_members` (relation N-N). Les groupes sont optionnellement affichés dans le tableau de comparaison.
- Les montures portent le `sensorType` (`FULL_FRAME` ou `APS_C`), utilisé pour calculer l'équivalent APS-C des objectifs et partagé avec les boîtiers interchangeables.
- **Filtres :** Filtres textuel (recherche dans label, marque, monture, options), par marque, monture, option, **type d'objectif** (`kind`: `"" | "prime" | "zoom"` via un select `Tous` / `Fixe` / `Zoom`), statut (favoris/à acheter/possédés), et quatre plages numériques via `DualRangeSlider` : focale min, focale max, ouverture à focale min, ouverture à focale max. Les plages utilisent le type `LensFilters` avec des champs `*Low`/`*High` (nombres), vérifiés de manière inclusive par `filterLenses()`. `filterLenses()` applique le filtre `Fixe` avec `isPrimeLens()` et `Zoom` comme inverse. Les sliders de focale utilisent désormais une plage UI **0–300** (plus 0–1000) ; la borne haute `300` est **ouverte** et signifie `300 mm et plus`, affichée `300+ mm` uniquement pour la valeur haute. La borne basse reste affichée comme une valeur simple (`300 mm`). Valeurs par défaut : focales [0, 300], ouvertures [1, 30]. Plus de filtre `sensorType`.
- **UI inventaire :** Tableau desktop, cartes mobile, graphique focale/ouverture, sélection et comparaison de 2 à 5 objectifs. Les tableaux/cartes/listes de paramètres n'utilisent plus de groupes de boutons inline répétés pour les actions par ligne : elles passent par le menu compact partagé `ActionMenu` (déclencheur ellipsis), ce qui réduit la largeur des colonnes d'actions et améliore le layout mobile. Les menus d'actions de tableaux ne dépendent plus d'un simple `z-index` local : le panel d'`ActionMenu` est rendu dans un portal vers `document.body` avec positionnement `fixed` calculé depuis le `getBoundingClientRect()` du trigger. La fermeture après clic sur un item du menu est désormais différée via `setTimeout(..., 0)` sur le `onClick` du panel, pour laisser les handlers des items et les soumissions de formulaire (ex. Modifier/Supprimer) s'exécuter correctement avant fermeture. Le tableau et les cartes affichent le type d'objectif dérivé (`Fixe` si `focalMinMm === focalMaxMm`, sinon `Zoom`), les badges de statut (`Favori`, `À acheter`, `Possédé`, `Retiré`) et évitent de répéter les plages identiques (ex. `7.8 mm`, `f/4`). Les prix et poids sont formatés de façon cohérente dans la liste, les cartes et la comparaison : `formatPrice()` groupe les milliers avec des espaces et affiche toujours 2 décimales ; `formatWeight()` groupe les milliers avec des espaces sans décimales. La valeur `0` est considérée comme valide et doit être affichée, pas traitée comme vide.
- **UI boîtiers :** inventaire boîtiers en tableau desktop et cartes mobile, avec filtres par texte, marque, monture, format capteur, type (`mirrorless`/`dslr`) et statut. Les actions par ligne/carte utilisent aussi `ActionMenu`. Le détail s'affiche dans `BodyDetailModal`; la création et l'édition utilisent `BodyForm` en modal. Les boîtiers portent : marque, nom/libellé, type, flag objectifs interchangeables, format capteur, monture partagée (nullable si boîtier à objectif fixe), mégapixels, ISO min/max, prix, poids, rafale, vidéo, autonomie, flags IBIS/double slot/tropicalisation/écran orientable, notes, statuts (`Favori`, `À acheter`, `Possédé`, `Retiré`).
- **Comparaison boîtiers :** sélection de 2 à 5 boîtiers via `toggleSelected` du contexte ; `BodyComparePopup` affiche la sélection en bas et ouvre une modale contenant `BodyCompareTable`.
- **UI accessoires :** le domaine `/accessories` est scindé en trois sous-catégories. `Sacs & poches` garde le tableau desktop/cartes mobile, les filtres texte/marque/type/statut/laptop/trépied, le détail via `AccessoryDetailModal` et la création/édition via `AccessoryForm`, avec affichage du poids conservé dans les listes/cartes. `Filtres & bagues` réutilise les mêmes actions/provider/référentiels, mais avec des filtres dédiés (texte, marque, type, statut, localisation, liaison), un formulaire spécialisé et un `FilterAssemblyAssistant` interactif en pile visuelle (Pillar Stack) : sélecteur d'objectif filtrant pour `(isOwned || isFavorite || isNextPurchase) && !retired` avec `useEffect` synchronisant `lensId`, puis une pile verticale de blocs (`LensBlock` en bas, `ConnectorLine` trait plein/pontillé selon fileté/magnétique, `AccessoryBlock` coloré par rôle avec bouton × pour démonter, `AddSlot` en haut avec bouton + ouvrant une liste de pièces compatibles et bouton "Monter") et une illustration SVG latérale `FilterStackIllustration` (style dessin d'enfant, rendu par rôle : bagues filetée/magnétique, disque filtre, pare-soleil, bloc générique) qui ne dessine qu'un seul connecteur entre chaque élément, le tout dans un layout flex horizontal `.filter-stack-layout`. Calcule tout en interne via `getActiveMountInterface`, `getMountedAssembly`, `getMountableAccessories`. Dans les présentations liste/carte de cette sous-catégorie, le texte de montage remonte la chaîne `mountedOnAccessoryId` jusqu'au premier objectif racine trouvé et affiche alors son libellé ; les chaînes cassées ou cycliques retombent sur `Monté`. Le poids n'y est pas affiché. `Autres accessoires` réutilise aussi le provider, les modales et les actions partagées, mais scope l'inventaire à `typeCategory="other"` avec des champs de formulaire conditionnels selon le `profile` du type choisi. Les accessoires `other` peuvent stocker des specs additionnelles `specCapacity`, `specFormat`, `specConnection`, `specCompatibility`, `specPower`, `specColorModes` et `specVariant`, affichées selon le profil dans les vues liste/carte/détail et dans le formulaire. `AccessoryLensReference` porte désormais les flags `isOwned`, `isFavorite`, `isNextPurchase`, `retired` peuplés depuis `listLenses()` dans `listAccessoryReferenceData()`. Pour les rôles filtres non `general`, le formulaire ne doit pas persister un nom/type libre : il dérive l'intitulé affiché depuis le rôle et les interfaces ; le rôle `general` conserve au contraire la saisie manuelle du type et du nom. Le formulaire accessoire soumet désormais explicitement le `<form>` via `onSubmit` + `new FormData(event.currentTarget)` ; en cas d'échec serveur/validation, les valeurs saisies restent en place et les erreurs sérialisées sont reformattées en texte lisible. Le composant interne `Field` alterne entre mode contrôlé (`value` + `onChange` si `value` fournie) et non contrôlé (`defaultValue` sinon) pour éviter l'avertissement React `defaultValue`+`value` simultanés.
- **Accessoires filtres/bagues :** les pièces de sous-catégorie `filter` portent `storageLocation`, montage direct sur objectif ou sur accessoire parent, interfaces arrière/avant (`none` / `threaded` / `magnetic`), rôle (`general` / `filter` / `adapter` / `hood`), `filterStrength` et `supportsMagneticHood`. Pour les interfaces `threaded` et `magnetic`, chaque endpoint de montage doit avoir un diamètre strictement positif ; le dépôt rejette aussi toute métadonnée d'interface de montage sur les accessoires hors sous-catégorie `filter`. Pour les adaptateurs, le couple rôle/interfaces pilote désormais aussi le type et le nom dérivés : `threaded→threaded` avec diamètres différents = `Bague de conversion`, `threaded→magnetic` ou `magnetic→magnetic` avec même diamètre = `Bague magnétique`, `threaded→magnetic` ou `magnetic→magnetic` avec diamètres différents = `Bague de réduction magnétique`, avec suffixe `avec pare-soleil` si le support du pare-soleil magnétique est coché. Les combinaisons `threaded→threaded` à diamètre identique et `magnetic→threaded` sont invalides. Les montages sont validés côté écriture pour empêcher les combinaisons incohérentes.
- Les objectifs ont un booléen `retired` (badge/libellé UI `Retiré`). Le formulaire objectif propose un import/aperçu client depuis un libellé puis expose aussi une case à cocher `Retiré` ; les marques, montures, focales, ouvertures max et options reconnues sont préremplies depuis les référentiels existants, puis l'aperçu est régénéré depuis les champs structurés. Après création ou modification réussie, le formulaire se ferme automatiquement ; il reste ouvert si l'action serveur échoue (ex. doublon détecté), et l'erreur est affichée dans un `<div className="form-error" role="alert">`.
- Les pages de paramètres (marques, montures, options, types d'accessoires, groupes d'options) chargent leurs propres données référentielles côté serveur — elles n'utilisent ni `LensProvider`, ni `BodyProvider`, ni `AccessoryProvider`. Leurs actions de ligne passent aussi par `ActionMenu` au lieu de boutons inline.
- Tests Vitest sur validation, utilitaires métier, parsing de libellés, dépôts SQLite, formulaires, page chart et popups/tableaux de comparaison. `ActionMenu.test.tsx` couvre le markup du menu partagé et la fermeture différée après clic d'item, mais un gap connu reste ouvert sur la couverture des régressions au niveau interaction complète (handlers d'items / soumissions de formulaire réelles). Côté accessoires : `AccessoryForm.test.tsx` couvre le rendu du composant `AccessoryForm`, les modes contrôlé/non contrôlé du composant interne `Field`, et a été stabilisé pour les initialisateurs paresseux de `useState` et les IDs canoniques de types `filter` ; `FilterAssemblyAssistant.test.ts` couvre le filtrage du sélecteur d'objectif (retirés exclus, non-flag exclus, next-purchase inclus). Côté boîtiers : validation, utilitaires `body-utils`, dépôt SQLite, `filterBodies()` (`BodyProvider.test.ts`), `BodyCompareTable`, des cas de présentation (`BodyPresentation.test.ts`), ainsi que validation des domaines partagés et assertions `BrandManager` pour le domaine `bodies`.

## Usage
- **Pour tout affichage lié aux objectifs :** utiliser `useLensContext()` depuis `@/components/lens/LensProvider` pour accéder aux filtres, sélection, masquage et données. Ne pas dupliquer l'état.
- **Pour tout affichage lié aux boîtiers :** utiliser `useBodyContext()` depuis `@/components/body/BodyProvider` pour accéder aux filtres, sélection, modales et données. Ne pas dupliquer l'état.
- **Pour tout affichage lié aux accessoires :** utiliser `useAccessoryContext()` depuis `@/components/accessory/AccessoryProvider` pour accéder aux filtres, modales détail/édition/création et données. Ne pas dupliquer l'état.
- **Pour modifier `/login` :** conserver `src/app/login/page.tsx` comme page serveur légère qui vérifie `hasValidSession()` et redirige les utilisateurs déjà authentifiés vers `/lenses`. Mettre toute logique interactive/état client du formulaire dans `src/components/auth/LoginForm.tsx` ; ne pas retransformer la page en composant client juste pour le formulaire.
- **Pour créer/modifier/supprimer un objectif :** utiliser les actions serveur dans `src/app/actions/lens-actions.ts` (`createLensAction`, `updateLensAction`, `deleteLensAction`). Elles appliquent auth, CSRF et `revalidatePath("/lenses", "layout")`. Une vérification de doublon est effectuée dans la transaction côté dépôt : si un objectif avec les mêmes `brandId`, `focalMinMm`, `focalMaxMm`, `maxApertureAtMinFocal`, `maxApertureAtMaxFocal`, `mountId` existe déjà, un `DuplicateLensError` est levé (message contenant le libellé existant). Exception actuelle : `updateLens()` autorise les modifications de champs non identitaires sur une ligne legacy déjà dupliquée, tant que cette mise à jour conserve exactement les mêmes champs d'identité ; en revanche, toute modification des champs d'identité qui entrerait en collision avec une autre ligne reste rejetée. L'erreur est attrapée par le formulaire client (`submitAndClose` try/catch) et affichée dans le `<div className="form-error">` — le formulaire reste ouvert pour permettre à l'utilisateur de corriger ou annuler.
- **Pour créer/modifier/supprimer un boîtier :** utiliser `src/app/actions/body-actions.ts` (`createBodyAction`, `updateBodyAction`, `deleteBodyAction`). Elles appliquent auth, CSRF et revalident `/bodies`.
- **Pour créer/modifier/supprimer un accessoire ou un type d'accessoire :** utiliser `src/app/actions/accessory-actions.ts` (`createAccessoryAction`, `updateAccessoryAction`, `deleteAccessoryAction`, `createAccessoryTypeAction`, `updateAccessoryTypeAction`, `deleteAccessoryTypeAction`). Elles appliquent auth, CSRF, et revalident le layout `/accessories` partagé par `/accessories`, `/accessories/filters` et `/accessories/others`, ainsi que la page de paramètres concernée pour les types.
- **Pour monter/démonter un accessoire filtre/bague :** utiliser `mountAccessoryOnLensAction(accessoryId, lensId)`, `mountAccessoryOnAccessoryAction(accessoryId, parentAccessoryId)` ou `unmountAccessoryAction(accessoryId)` depuis `src/app/actions/accessory-actions.ts`. Elles retournent `{ success, error }`, appliquent `assertAuthenticated()`, et revalident `/accessories`.
- **Pour créer/modifier/supprimer une marque, monture, option ou groupe d'options :** utiliser les actions correspondantes (`createBrandAction`, `updateMountAction`, `deleteOptionAction`, `createOptionGroupAction`, `setOptionGroupMembersAction`, etc.). Les actions de marques revalident `/settings/brands`, `/lenses`, `/bodies` **et** `/accessories` car les marques sont partagées entre domaines. Les montures sont partagées entre objectifs et boîtiers.
- **Les pages de paramètres** (`/settings/brands`, `/settings/mounts`, `/settings/options`, `/settings/options/groups`, `/settings/accessory-types`) chargent leurs propres données référentielles côté serveur — elles n'utilisent ni `LensProvider`, ni `BodyProvider`, ni `AccessoryProvider`.
- **Le zoom D3.js dans LensChart :** navigation zoom (pas loupe). Les traits utilisent `vectorEffect="non-scaling-stroke"` pour une épaisseur constante. Les rayons des cercles sont contre-échelonnés via la fonction `counterScaleCircles(root, k)`. Le zoom/pan est initialisé une seule fois dans un `useEffect(() => {}, [])` ; la transformation D3 est appliquée directement au DOM — aucune re-render React pendant les gestes. L'instance D3 zoom est stockée dans `zoomRef`. `counterScaleCircles` est appelée depuis le handler D3 (tous les événements de zoom) et depuis un `useLayoutEffect` qui re-synchronise en amont du rendu quand `lenses` ou `selectedIds` changent (pour éviter un flash de rayons non contre-échelonnés après un re-render React en zoom). Ne pas remplacer ce pattern par un état React.
- **Contre-échelle des cercles :** une seule constante contrôle le comportement : `MIN_CIRCLE_R` (2). La formule appliquée est `Math.max(MIN_CIRCLE_R, base / k)` avec un plancher. Les cercles restent à une taille visuelle constante à tout niveau de zoom (rayon visuel = `k × (base/k)` = **base constant**). Les cercles portent un attribut `data-base-r` contenant le rayon de base. Pour modifier le rendu du zoom, éditer le `MIN_CIRCLE_R` et/ou `base / k` dans `counterScaleCircles`.
- **Le graphique montre uniquement les objectifs cochés :** `LensChart` reçoit `checkedLenses` (filtrés depuis `initialLenses` par `checkedIds`, état local `useState` dans `LensChartPage`, pas de limite). L'état initial coche uniquement les objectifs possédés et non retirés (`isOwned && !retired`) ; les autres restent disponibles dans la sidebar et peuvent être cochés manuellement. Les filtres (`LensFiltersBar`) et `hiddenIds` n'affectent que la liste de la sidebar (pour faciliter le choix), pas le jeu d'objectifs disponibles pour la sélection. Le clic sur une ligne/cercle du graphique appelle `onToggleSelected` (ajoute ou retire de la comparaison, max 5). Les objectifs en comparaison sont surlignés en orange via `selectedIds`. La sidebar a deux contrôles par ligne : **checkbox** (`toggleChecked`, visibilité, pas de limite) et **bouton étoile** (`toggleSelected`, comparaison, max 5). Le clic sur le libellé déclenche aussi `toggleChecked`.
- **Bouton Réinitialiser :** utiliser `resetZoom()` (mémorisée via `useCallback`) pour une transition fluide de 500ms vers `d3.zoomIdentity`. Le bouton est toujours rendu, même dans l'état vide.
- **Filtres plages numériques :** réutiliser `DualRangeSlider` (props : `min`, `max`, `low`, `high`, `onChange`, `step`, `label`, `formatValue`, avec `formatLowValue` / `formatHighValue` optionnels si les deux bornes doivent être affichées différemment). Les contraintes `low ≤ high` sont assurées par les handlers `handleLowChange`/`handleHighChange` qui clampent au moment du changement. Pour la focale, réutiliser les constantes partagées de `src/lib/lens/filter-constants.ts` et conserver la sémantique actuelle : borne haute `300` affichée `300+ mm`, borne basse inchangée (`300 mm`).
- **`filterLenses()` (pure function) :** importable depuis `LensProvider.tsx` pour tests et usage direct. Vérifie chaque filtre de manière inclusive : la valeur de l'objectif doit être entre `*Low` et `*High` inclus. Le filtre `kind` accepte `prime` / `zoom` et s'appuie sur `isPrimeLens()` (zoom = inverse). Les champs `sensorType`, `focalMin`, `focalMax`, `maxAperture` ont été supprimés de `LensFilters` ; utiliser `kind` et les paires `*Low`/`*High` à la place.
- Réutiliser `lens-repository.ts` pour toute lecture/écriture SQLite au lieu d'accéder directement à la base depuis les composants.
- Réutiliser `body-repository.ts` pour toute lecture/écriture SQLite liée aux boîtiers. Les labels sont dérivés côté dépôt via `normalizeBodyInput()` / `generateBodyLabel()` ; ne pas persister de libellé libre.
- Réutiliser `accessory-repository.ts` pour toute lecture/écriture SQLite liée aux accessoires. Les labels sont dérivés côté dépôt via `normalizeAccessoryInput()` / `generateAccessoryLabel()` ; ne pas persister de libellé libre.
- Pour toute logique de type intégré accessoire, réutiliser `src/lib/accessory/accessory-type-ids.ts` et les IDs `ACCESSORY_TYPE_IDS` au lieu de dépendre des libellés affichés. Les noms des types intégrés peuvent être renommés dans les référentiels, mais la résolution des types filtres dérivés côté client (`resolveFilterAccessoryTypeId`) et côté serveur (`deriveAccessoryPersistenceInput`) doit rester basée sur les IDs canoniques. Même règle pour les types/profils intégrés de `other` : s'appuyer sur les IDs/profils stables plutôt que sur les libellés.
- **Déploiement conteneur :** privilégier le flux GitHub Actions -> GHCR -> Portainer avec `docker-compose.portainer.yml`. `publish-image.yml` publie toujours un tag `sha-<commit>` réutilisable par `PHOTOPARK_IMAGE`, et ne pousse `latest` que pour les pushes sur `main`; conserver cette convention au lieu de cibler `latest` pour les déploiements stables. Pour Portainer, fournir `PHOTOPARK_IMAGE` (préférer un tag SHA ; si disponible, un digest est plus immuable qu'un tag), créer le volume Docker externe `photopark-data` avant le premier déploiement, fournir au minimum `APP_PASSWORD_HASH` + `SESSION_SECRET`, et conserver la configuration d'accès GHCR requise côté Portainer. `docker-compose.yml` reste le fallback local/manuelle build-based. Il n'existe plus de workflow de déploiement séparé ni de script wrapper : pour un déploiement manuel, utiliser directement les commandes `docker compose` adaptées au fichier compose choisi.
- Réutiliser `parseLensFormData` / `lensSchema` pour le formulaire objectif, y compris le booléen `retired` ; les nombres peuvent utiliser un point ou une virgule.
- Réutiliser `parseBodyFormData` / `bodySchema` pour le formulaire boîtier ; les nombres peuvent utiliser un point ou une virgule, les champs vides deviennent `null`, `mountId` est requis seulement si `isInterchangeableLens === true`.
- Réutiliser `parseAccessoryFormData` / `accessorySchema` pour le formulaire accessoire ; les nombres peuvent utiliser un point ou une virgule, et les champs vides deviennent `null`. Pour les accessoires `filter` dont le rôle n'est pas `general`, laisser le serveur dériver `typeId` et le nom persisté depuis le rôle, les interfaces et les diamètres au lieu de forcer une saisie libre ; pour `filterRole="general"`, conserver les champs manuels `typeId` + `name`. Pour les accessoires `other`, laisser le `profile` du type piloter quels champs `spec*` sont visibles et persistés ; ne pas déduire cette logique depuis le libellé du type. Conserver le pattern actuel de soumission explicite `onSubmit` → `new FormData(event.currentTarget)` dans `AccessoryForm` pour que les échecs de soumission n'effacent pas l'état local et que les erreurs sérialisées soient affichées dans un format lisible. Le composant interne `Field` alterne entre mode contrôlé (`value` + `onChange` si `value` fournie) et non contrôlé (`defaultValue` sinon) : réutiliser ce pattern pour tout champ `<input>` d'AccessoryForm afin d'éviter l'avertissement React lorsque les deux props sont présentes simultanément.
- Pour créer/modifier des objectifs, fournir des IDs de référentiels (`brandId`, `mountId`, `optionIds`) ; marque, monture, capteur, options, label et équivalents APS-C sont normalisés côté dépôt.
